## What just shipped (v0.10.0 adoption, PR #3)

Single commit `74057b0`. Migrated three `manager._app_config = ...  # type: ignore[attr-defined]` private-attribute writes to PluginForge v0.10.0's public `merge_app_config()` entry point. Consolidated the duplicated `_refresh_manager_app_config()` helpers (one each in `settings.py` and `plugin_install.py`) into a single `config_overlay.refresh_manager_overlay()` function.

### Backend changes

- `backend/pyproject.toml`: pluginforge `^0.9.0` -> `^0.10.0`, lock regenerated.
- `backend/app/config_overlay.py`: two new module-level helpers.
  - `read_user_overlay() -> dict[str, Any]`: one-line wrapper around `_read_yaml(_user_app_path())`. Returns the user-overlay layer alone (no project YAML baseline).
  - `refresh_manager_overlay(manager: PluginManager | None, *, notify: bool = True) -> list[PluginError]`: encapsulates `manager.reload_config()` + `manager.merge_app_config(read_user_overlay(), notify=notify)`. Accepts `None` (returns `[]`) so router-level callers with `_manager: PluginManager | None` don't need a local None guard.
- `backend/app/main.py`: `_sync_manager_with_overlay()` body shrunk from 15 lines to a one-line delegate to `refresh_manager_overlay(manager, notify=False)` (pre-discovery, no active plugins).
- `backend/app/routers/settings.py`: deleted the local `_refresh_manager_app_config()` helper; 5 callsites (147, 209, 392, 452, 470) now call `config_overlay.refresh_manager_overlay(_manager)`. `notify=True` default fires `on_config_changed` on active plugins.
- `backend/app/routers/plugin_install.py`: deleted the mirrored helper; 2 callsites (224, 253) migrated the same way.

## Open: Deploy Docs workflow has never succeeded

Surfaced at the end of the v0.10.0 session via the GitHub Actions notification on the PR #3 merge.

**Symptom:** `scripts/generate_mkdocs_nav.py` exits with `ValueError: marker not found: '# AUTO-GENERATED-NAV-START (do not edit; regenerate via make sync-mkdocs-nav)'`.

**Root cause:** `mkdocs.yml` doesn't have the marker pairs the script expects. `scripts/generate_mkdocs_nav.py` (from `a564d82` initial version) was authored upstream (adaptive-learner) where the markers presumably existed; the template inherited the script without inheriting the markers. The `nav:` block (currently at lines 88-122) and the `nav_translations:` block (currently at lines 47-67 inside the i18n plugin config) are entirely hand-maintained.

**History:** every Deploy Docs run since `a564d82` (initial commit, 2026-05-17) has failed with the same error. Verified via `gh run list --repo astrapi69/pluginforge-app-template --workflow docs.yml`:

```
26236583021  failure  Merge PR #3 (v0.10.0)       2026-05-21T15:43Z
26219221105  failure  Merge PR #1 (v0.9.0)        2026-05-21T10:01Z
26168722757  failure  docs: comprehensive README  2026-05-20T14:21Z
25997067694  failure  initial version             2026-05-17T16:58Z
```

The v0.10.0 PR didn't cause it; it just triggered another run by touching files under `docs/help/**`.

**What the fix needs to do:**

1. Insert `# AUTO-GENERATED-NAV-START` / `-END` marker pair in `mkdocs.yml` `nav:` block, after `- Home: de/index.md` (the Home line stays hand-managed per the script's docstring).
2. Insert `# AUTO-GENERATED-NAV-TRANSLATIONS-START` / `-END` marker pair in the English locale's `nav_translations:` block inside the i18n plugin config, indented to 8 spaces.
3. Run `poetry run python scripts/generate_mkdocs_nav.py` once to populate the markers from `docs/help/_meta.yaml` (the single source of truth per the script's docstring).
4. Commit the result.

**`docs/help/_meta.yaml` exists** - the script has its input. Whether the script's regenerated output matches the current hand-maintained nav is the open question. The v0.30.0 docs+i18n drift audit (referenced in the script's docstring) is the original motivation for the script existing.

**Three scopes proposed at the end of the v0.10.0 session, awaiting decision:**

1. **Minimum**: add the marker pairs around the existing content unchanged, regenerate, commit. If the script's output happens to match the current nav, single mechanical PR. If it differs, surface the diff and decide.
2. **Audit-first** (recommended): run the script locally, diff the generated content against the current `mkdocs.yml`, report what would change, then decide whether to accept the regen or update `_meta.yaml` first.
3. **Defer**: open a tracking issue. The docs site has been broken since day one of the template; one more release cycle doesn't make it worse. Pair with the next docs-site work.
